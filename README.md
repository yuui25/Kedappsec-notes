# keda-lab

このリポジトリは、Web／ネットワーク／クラウド／ID連携を含む安全性検証を **「境界→観測→差分→結論」** の型で回すための、知識・手順・記録・検証環境をひとまとめにした作業台です。  
「その場の思いつき」ではなく、毎回同じ粒度・同じ根拠で判断できることを目的にしています。

---

## 使い方の基本サイクル（迷ったらここに戻る）
1. **理解する（何を見れば良いか）**  
   `01_topics/` を開いて、対象の仕組み・境界（資産／信頼／権限）・観測ポイントを押さえる。
2. **実行する（何からやるか）**  
   `02_playbooks/` を開いて、順番・分岐・最小証跡を固定して進める。
3. **記録する（後で説明できる形にする）**  
   `03_cases/` に、事実／仮説／検証／証跡／結論（成立・不成立・不明）を残す。
4. **再現する（確度を上げる）**  
   `04_labs/` を使って、安全な環境で再現し、成立条件と影響を切り分ける。
5. **整える（同じ粒度で増やす）**  
   `99_templates/` を使って、ドキュメントを同じ型で追加し、index（入口）から辿れるようにする。

---

## フォルダ構成と「見るタイミング」
| フォルダ | 役割 | いつ見るか | 使い方のコツ |
|---|---|---|---|
| `01_topics/` | 仕組み・境界・成立条件・観測ポイント（判断材料） | 「何が起きている？」「どこが境界？」を整理したいとき | “用語説明”より **境界と差分**を読みに行く |
| `02_playbooks/` | 実施手順（順番・分岐・最小証跡） | 「今日は何からやる？」を迷わず始めたいとき | index から該当テーマを選び、手順を崩さない |
| `03_cases/` | 作業ログ（事実・仮説・検証・証跡・結論） | 作業の開始時・終了時、結論をまとめるとき | “事実”と“解釈”を分けて書く（後で強い） |
| `04_labs/` | 検証環境（観測点・到達性・教材・ログ） | 本番相当で確信が持てない／影響の切り分けが必要なとき | 観測点（Proxy/HAR/pcap/ログ）を固定して再現する |
| `99_templates/` | テンプレ（同じ粒度で書くための型） | 新しいファイルを追加するとき／書き方がブレそうなとき | まずテンプレで作り、後から中身を埋める |

## よくある利用シーン別の動き方
### 1) 新しい対象に着手する（入口と境界を固める）
- `02_playbooks/` の index から該当する入口手順を選び、最小の観測（HTTP/証明書/JS/クラウド露出など）を揃える  
- その結果を `03_cases/` に「事実」として残す  
- “意味づけ”が必要になったら `01_topics/` に戻り、境界（資産／信頼／権限）に落とす

### 2) 認証・認可・API・入力・設定など特定テーマを深掘りする
- まず `01_topics/` で「観測ポイント」と「差分の作り方（何を変えるか）」を確認  
- 次に `02_playbooks/` に従って実施し、必ず `03_cases/` に証跡を残す  
- “成立条件が曖昧”な場合は `04_labs/` で再現して確度を上げる

### 3) 影響説明（結論）を固めたい
- `03_cases/` に、以下のセットが揃っているか確認する  
  - 前提（誰の状態／どの条件）  
  - 手順（最小）  
  - 証跡（リクエスト/レスポンス、時刻、ID、ログ保存先）  
  - 結果（差分）  
  - 結論（成立・不成立・不明）と、不明なら次の観測点  
- 足りない場合は `04_labs/` で再現し、差分が取れる状態にしてからケースへ戻す

---

## 注意事項（必須）
- 本リポジトリは、**許可された範囲**での安全性検証・学習・再現を前提としています。対象範囲・規約・法令を必ず確認してください。
- 結論は推測で断言せず、証跡（観測結果）に基づいて記述してください。不明な場合は“不明”とし、次に必要な観測点を明記してください。

## はじめに見る場所（最短導線）
- 目的と全体像：`01_topics/*/00_index.md`
- すぐ始める手順：`02_playbooks/00_index.md`
- 作業ログの母艦：`03_cases/00_index.md`
- 検証環境の入口：`04_labs/00_index.md`
- 書き方の型：`99_templates/`
