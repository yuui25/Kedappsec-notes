# 00_index.md

（先頭要約：ガイドライン対応の位置づけ）
- ASVS：各プレイブックは、ASVSの要求（認証・認可・秘密情報・監査など）を“実務の意思決定”に落とすための導線として使う
- WSTG：テスト観点を「やる/やらない」「どこを見る/どこまでやる」に変換し、観測→意味→次の一手に接続する
- PTES：各プレイブックはPTESフェーズを跨いで使う（IG→VA→Exploitation→Post）ため、前後のつながりを短く明示する
- ATT&CK：分類のためではなく、攻撃者目的（Discovery/Credential/Lateral等）と“境界崩壊”の説明補助として使う

## 目的（このプレイブック群で到達する状態）
- 02_playbooks は「手順書」ではなく、現場で迷わないための **意思決定の導線** を提供する。
- 01_topics で作った技術ユニット（観測→解釈→利用）を、現場の進め方（順序・分岐・最小証跡）に落とし込む。
- “やることの網羅”ではなく、次を実現する。
  - 優先度が付く（どれから深掘りするか）
  - 分岐が明確（条件AならA、違えばB）
  - 証跡が残る（再現性と説明力が上がる）

## 前提（対象・範囲・想定）
- 対象：許可範囲のASM/OSINT、Web、NW、SaaSを含むペネトレーションテスト/脆弱性深掘り。
- 想定：
  - Web/NW=5:5 を維持し、片方に偏らない
  - ツールの操作説明は最小限にし、必ず「意味→判断→次の一手」を主役にする
- 依存（接続先）：
  - 01_topics：各プレイブックの根拠（観測ポイント、状態、分岐）
  - 04_labs：観測・改変・証跡取得の基盤（Proxy、pcap、巻き戻し）
  - 03_cases：実世界事例を“教材”として参照（要約ではなく技術ユニットへ接続）

## プレイブックの使い方（現場で迷わない運用）
### 使い方の原則
- プレイブックは「読む」ものではなく「回す」もの
- 1回で全部やらない。必ず以下の順で最小化する
  1) 入口と境界を確定（資産/信頼/権限）
  2) 観測して状態を言語化（何が言える/言えない）
  3) 分岐で次の一手を選ぶ（A/B）
  4) 証跡を残す（再現性）

### 最小証跡（共通で残すもの）
- 条件：経路、主体（ユーザー/権限）、対象、範囲
- 観測：主要な応答/ログ（必要ならHAR/pcap）
- 解釈：状態（境界がどうなっているか）
- 次の一手：A/B分岐の選択理由

## プレイブック作成の品質ルール（indexとしての最低限）
- 1ファイルは「導線（順序・分岐・最小証跡）」に集中し、説明の詳細は 01_topics へ寄せる
- コマンド例は最小限（例示）。必ず「その出力の意味」と「次の判断」を書く
- すべてのプレイブックで、最後に「関連topics/labs/cases」の内部リンクを置く

## ガイドライン対応（ASVS / WSTG / PTES / MITRE ATT&CK：毎回）
- ASVS：
  - プレイブックはASVS項目の“貼り付け”ではなく、実務の判断（境界・権限・秘密情報・監査）に落とすための導線として使う
- WSTG：
  - テスト観点を「観測→意味→次の一手」に変換し、どこまでやるか（範囲/影響）を制御する
- PTES：
  - 各プレイブックはフェーズを跨いで使うため、冒頭で該当フェーズと前後接続を短く明示する
- MITRE ATT&CK：
  - 戦術は“作戦の説明補助”として用い、境界崩壊（Discovery/Credential/Lateral等）を言語化する
